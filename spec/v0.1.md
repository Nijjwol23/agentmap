## Abstract

The Web2Agent (W2A) Protocol defines a standard for websites to
declare their capabilities to AI agents. A conforming website
serves a machine-readable manifest file at a well-known URI. A
conforming agent reads this manifest before interacting with the
site, enabling direct, efficient, intent-driven access without
blind crawling.

W2A is designed to complement the Agent2Agent Protocol (A2A) and
the Model Context Protocol (MCP). A W2A manifest is a valid A2A
AgentCard profile. Any A2A-compliant client can read a W2A
manifest without modification.

## 1. Motivation

AI agents that visit websites today have no standard way to
discover what a site can do. An agent attempting to buy a product,
read an article, or book a service must crawl the site's pages,
parse HTML, infer structure, and guess at available actions. This
is wasteful, error-prone, and places unnecessary load on origin
servers.

The web has solved analogous problems before:

- `robots.txt` (1994) told crawlers what not to index
- `sitemap.xml` (2005) told crawlers where pages are
- W2A `agents.json` (2026) tells agents what a site can **do**

W2A does not replace human-facing HTML. It provides a parallel,
machine-readable declaration of capabilities that AI agents can
read once and act on directly.

## 2. Terminology

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**,
**SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**,
and **OPTIONAL** in this document are to be interpreted as described
in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119).

**Site** — any web-accessible server that serves content or
provides actions to human or machine clients.

**Agent** — any automated software client that reads W2A manifests
and acts on declared capabilities.

**Manifest** — the `agents.json` file served by a site conforming
to this specification.

**Capability** — a single declared action a site exposes to agents.

**Intent** — a plain-English string describing what a capability
does, written for an agent to read and reason about.

---

## 3. Manifest file

A conforming site **MUST** serve a valid W2A manifest as a JSON
file at the well-known URI:

```
/.well-known/agents.json
```

This URI is registered under the `.well-known` scheme defined in
[RFC 8615](https://www.rfc-editor.org/rfc/rfc8615).

The file **MUST** be:

- Valid JSON (RFC 8259)
- Encoded in UTF-8
- Served with `Content-Type: application/json`
- Accessible via HTTP GET without authentication
- Served with `Access-Control-Allow-Origin: *` to permit
  cross-origin agent requests

The file **SHOULD** be:

- Served with `Cache-Control: public, max-age=3600`
- Updated whenever the site's capabilities change
- Kept under 100KB in size

---

## 4. Discovery

Agents discover the manifest by sending an HTTP GET request to
`/.well-known/agents.json` on the target origin before interacting
with any other resource on that site.

```
GET /.well-known/agents.json HTTP/1.1
Host: example.com
Accept: application/json
Agent-Intent: <intent>          (OPTIONAL — see section 12)
Agent-Identity: <identity>      (OPTIONAL — see section 12)
```

A `404 Not Found` response indicates the site does not support W2A.
Agents **SHOULD** fall back to standard crawling behaviour in this
case.

A `200 OK` response with a valid manifest indicates full W2A
support. Agents **SHOULD** use the manifest as the primary source
of truth for capability discovery.

## 5. Schema

### 5.1 Top-level object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `w2a` | string | **yes** | Spec version. MUST be `"1.0"` for this version. |
| `site` | Site object | **yes** | Metadata about the site. |
| `capabilities` | Capability[] | **yes** | List of declared capabilities. MUST contain at least one entry. |
| `policies` | Policy object | no | Access control and rate limiting. |
| `federation` | Federation[] | no | References to external manifests. |
| `a2a_profile` | A2A profile object | no | A2A AgentCard compatibility block. |

### 5.2 Site object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | **yes** | Human-readable name of the site. |
| `type` | string | **yes** | Site category. See permitted values below. |
| `language` | string | no | Primary language. BCP 47 tag e.g. `"en"`, `"fr"`. Defaults to `"en"`. |
| `description` | string | no | One-sentence description of the site for agent context. |

**Permitted `type` values:**

| Value | Description |
|-------|-------------|
| `ecommerce` | Sells physical or digital products |
| `blog` | Publishes articles or written content |
| `saas` | Software-as-a-service product |
| `marketplace` | Multi-vendor buying and selling platform |
| `media` | News, video, audio, or multimedia content |
| `directory` | Listings, reviews, or reference data |
| `api` | Primarily an API with no human-facing UI |
| `other` | Any site type not covered above |

### 5.3 Capability object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | **yes** | Unique identifier within this manifest. Lowercase, underscores only. e.g. `"search_products"` |
| `intent` | string | **yes** | Plain-English description of what this capability does. Written for an agent to read. |
| `action` | string | **yes** | HTTP method and path. Format: `"METHOD /path"`. e.g. `"GET /api/search"` |
| `input` | Input object | no | Named input parameters the agent must or may supply. |
| `output` | Output object | no | Named output fields the agent can expect in the response. |
| `auth` | string | **yes** | Authentication requirement. See section 7. |
| `description` | string | no | Additional context beyond `intent`. For complex capabilities. |

**`id` format:** lowercase letters, digits, and underscores only.
MUST be unique within the manifest. MUST NOT be empty.

**`intent` guidance:** Write the intent as a plain-English sentence
an agent reads to decide whether to use this capability. Be
specific. Avoid technical jargon. Good: `"find products by keyword
or category"`. Poor: `"calls the search endpoint"`.

**`action` format:** `METHOD /path` where METHOD is one of `GET`,
`POST`, `PUT`, `PATCH`, `DELETE`. Path parameters use colon
notation: `/api/products/:id`.

### 5.4 Input and output objects

Input and output objects are plain JSON objects where each key is a
parameter name and each value is a type string.

```json
{
  "q": "string",
  "category": "string?",
  "limit": "int?"
}
```

Parameter names **MUST** be lowercase with underscores.
Type values **MUST** be drawn from the type system defined in
section 6.

### 5.5 Policy object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `rate_limit` | string | no | Maximum request rate. Format: `"N/unit"` where unit is `sec`, `min`, or `hour`. e.g. `"60/min"` |
| `allowed_agents` | string[] | no | Agent identifiers permitted to access capabilities. `["*"]` permits all agents. Defaults to `["*"]` if omitted. |
| `blocked_agents` | string[] | no | Agent identifiers explicitly denied. Evaluated after `allowed_agents`. |
| `require_identity` | boolean | no | If `true`, agents MUST send `Agent-Identity` header. Defaults to `false`. |

### 5.6 Federation object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `role` | string | **yes** | Semantic role of the referenced manifest. e.g. `"shipping"`, `"payments"`, `"auth"` |
| `ref` | string | **yes** | Full URL to the external `agents.json` file. MUST use HTTPS. |
| `description` | string | no | Human-readable description of what the referenced site provides. |

### 5.7 A2A profile object

Enables A2A-compliant agents to treat this manifest as a valid
AgentCard. See section 10 for full compatibility details.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | **yes** | Agent name as it appears in the A2A ecosystem. |
| `url` | string | **yes** | Full URL to this manifest. MUST match the serving URL. |
| `version` | string | **yes** | Version string e.g. `"1.0"` |
| `description` | string | no | Agent description for A2A discovery. |

## 6. Type system

Input and output fields use a simple string type notation:

| Type | Description |
|------|-------------|
| `string` | Required UTF-8 string |
| `string?` | Optional UTF-8 string |
| `int` | Required integer |
| `int?` | Optional integer |
| `float` | Required floating point number |
| `float?` | Optional floating point number |
| `bool` | Required boolean (`true` or `false`) |
| `bool?` | Optional boolean |
| `object` | Required JSON object (untyped) |
| `object?` | Optional JSON object |
| `string[]` | Required array of strings |
| `int[]` | Required array of integers |
| `object[]` | Required array of JSON objects |

Arrays of optional items are not supported in v0.1. Optional arrays
use the `?` suffix: `string[]?`.

Custom named types (e.g. `Product[]`) **MAY** be used in output
objects for documentation clarity. Agents **SHOULD** treat unknown
type names as `object` or `object[]`.

---

## 7. Authentication

The `auth` field on each capability declares what authentication
the agent must provide to call that action.

| Value | Description |
|-------|-------------|
| `none` | No authentication required. Publicly accessible. |
| `session` | Requires an active user session (cookie-based). Agent must have established a session on behalf of a user. |
| `bearer` | Requires a Bearer token in the `Authorization` header. |
| `apikey` | Requires an API key. Delivery mechanism (header, query param) is site-specific. |
| `oauth2` | Requires OAuth 2.0 flow. |

Sites **SHOULD** document authentication setup in their developer
docs and link to it from the `description` field on affected
capabilities.

---

## 8. Access policies

When `allowed_agents` is set to a list of named identifiers, only
those agents may access the capabilities in the manifest. All
others receive `403 Forbidden`.

Agent identifiers are strings matching the `Agent-Identity` request
header value (see section 12).

When both `allowed_agents` and `blocked_agents` are present,
`blocked_agents` takes precedence. A blocked agent is denied even
if it appears in `allowed_agents`.

`rate_limit` is advisory in v0.1. Sites **SHOULD** enforce it
server-side but agents **SHOULD** also respect it proactively.

---

## 9. Federation

Federation allows a site to declare that some capabilities are
served by a third-party agent. An agent traversing a federated
manifest **SHOULD** fetch and read the referenced manifest before
attempting to invoke capabilities in that role.

```json
{
  "federation": [
    {
      "role": "shipping",
      "ref": "https://fedex.com/.well-known/agents.json",
      "description": "Shipping and tracking capabilities"
    }
  ]
}
```

Agents **MUST NOT** follow federation references more than two
levels deep to prevent cycles.

Referenced manifests **MUST** be fetched over HTTPS. Agents
**SHOULD** cache federated manifests independently with their own
`Cache-Control` headers.

## 10. A2A compatibility

A W2A manifest with an `a2a_profile` block is a valid A2A
AgentCard. The mapping is as follows:

| W2A field | A2A AgentCard field |
|-----------|---------------------|
| `a2a_profile.name` | `name` |
| `a2a_profile.url` | `url` |
| `a2a_profile.version` | `version` |
| `a2a_profile.description` | `description` |
| `capabilities[].id` | `skills[].id` |
| `capabilities[].intent` | `skills[].description` |
| `policies.rate_limit` | (advisory) |

A2A clients discovering agents via `/.well-known/agent.json` (the
A2A discovery path) **MAY** also check `/.well-known/agents.json`
(the W2A path) if the A2A path returns 404.

W2A does not require A2A compatibility. The `a2a_profile` block is
**OPTIONAL**. Sites that only want to support W2A-native agents
**MAY** omit it.

## 11. HTTP response requirements

### Serving the manifest

```
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Access-Control-Allow-Origin: *
Cache-Control: public, max-age=3600
ETag: "<hash-of-content>"
```

### Error responses

| Status | Meaning |
|--------|---------|
| `200 OK` | Valid manifest returned |
| `404 Not Found` | Site does not support W2A |
| `403 Forbidden` | Agent is blocked by access policy |
| `429 Too Many Requests` | Rate limit exceeded |
| `503 Service Unavailable` | Manifest temporarily unavailable |

### Conditional requests

Sites **SHOULD** support `If-None-Match` conditional requests using
the `ETag` header. Agents **SHOULD** send `If-None-Match` on
subsequent requests to avoid re-fetching unchanged manifests.

## 12. Agent request headers

Agents **MAY** send the following headers when fetching the
manifest or calling capabilities:

| Header | Description |
|--------|-------------|
| `Agent-Intent` | A short string describing what the agent intends to do on this site. e.g. `"purchase"`, `"read"`, `"search"` |
| `Agent-Identity` | A unique identifier for the agent. Format: `"org/agent-name"` e.g. `"anthropic/claude"` |
| `Agent-Version` | Version of the agent. e.g. `"3.5"` |
| `Agent-W2A` | W2A spec version the agent implements. MUST be `"1.0"` for this spec. |

Sites **MAY** use `Agent-Identity` to personalise the manifest
response — returning a richer manifest to verified agents and a
basic manifest to anonymous agents.

Sites **MUST NOT** require these headers for public capabilities
(`auth: none`). They are informational and **OPTIONAL**.

---

## 13. Versioning

The `w2a` field in the top-level object declares the spec version.
This document defines version `"1.0"`.

Agents encountering an unknown version **SHOULD** attempt to parse
the manifest anyway, as future versions are designed to be
backwards-compatible with v0.1.

Breaking changes will increment the major version number and will
not be introduced without a transition period.

---

## 14. Security considerations

**Manifest authenticity.** W2A manifests are served over HTTPS.
Agents **MUST NOT** read manifests over plain HTTP. Agents
**SHOULD** verify the manifest is served from the same origin as
the site they intend to interact with.

**Injection.** The `intent` field is written by site operators and
read by agents. Site operators **MUST NOT** use the intent field to
inject instructions that could manipulate agent behaviour beyond
the scope of the declared capability. Agents **SHOULD** treat
intent strings as informational, not as executable instructions.

**Overly broad permissions.** Sites **SHOULD NOT** declare
capabilities with `auth: none` that expose sensitive data or
destructive actions. Capabilities that modify state **SHOULD**
require at minimum `session` or `bearer` auth.

**Federation trust.** Agents **SHOULD NOT** automatically trust
federated manifests to the same degree as the origin manifest.
Federated manifests are third-party declarations and **SHOULD** be
treated accordingly.

**Rate limiting.** Agents **MUST** respect `rate_limit` policies.
Ignoring rate limits is a violation of this specification.

## 15. Examples

### Minimal valid manifest

```json
{
  "w2a": "1.0",
  "site": {
    "name": "Example Site",
    "type": "other"
  },
  "capabilities": [
    {
      "id": "get_info",
      "intent": "get general information about this site",
      "action": "GET /api/info",
      "auth": "none"
    }
  ]
}
```

        "items": "object[]",

      "id": "get_product",
      "intent": "get full details for a specific product including stock level",
      "action": "GET /api/products/:id",
      "input": { "id": "string" },
      "output": {
        "id": "string",
        "name": "string",
        "price": "float",
        "currency": "string",

      "input": {
        "cart_id": "string",

      "output": {
        "order_id": "string",

## 16. Conformance

### Site conformance

A **conforming site** MUST:

- Serve a valid `agents.json` at `/.well-known/agents.json`
- Return `Content-Type: application/json`
- Include `Access-Control-Allow-Origin: *`
- Declare at least one capability
- Use only permitted `type` values in the `site.type` field
- Use only permitted `auth` values in capability `auth` fields
- Ensure all `id` values are unique within the manifest

A **conforming site** SHOULD:

- Return appropriate HTTP status codes (section 11)
- Support `ETag` and conditional requests
- Enforce declared `rate_limit` policies server-side
- Keep the manifest current when capabilities change

### Agent conformance

A **conforming agent** MUST:

- Fetch `/.well-known/agents.json` before interacting with a site
- Respect `blocked_agents` policies
- Respect `rate_limit` policies
- Fetch manifests over HTTPS only
- Not follow federation references more than two levels deep

A **conforming agent** SHOULD:

- Send `Agent-W2A: 1.0` header
- Cache manifests according to `Cache-Control` headers
- Send `If-None-Match` on subsequent manifest requests
- Fall back to standard crawling when manifest returns 404
